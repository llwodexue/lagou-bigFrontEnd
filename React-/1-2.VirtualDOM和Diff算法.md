# Virtual DOM 和 Diff 算法

## 什么是 Virtual DOM

Virtual DOM（虚拟 DOM）是 React 的核心概念之一。它是一个轻量级的 JavaScript 对象，用来描述真实的 DOM 结构。

### 为什么需要 Virtual DOM

在传统的 DOM 操作中，每次修改页面都需要直接操作 DOM 节点，而 DOM 操作是非常昂贵的。Virtual DOM 的出现解决了以下问题：

1. **性能优化**：批量更新 DOM，减少直接操作 DOM 的次数
2. **跨平台能力**：Virtual DOM 是纯 JavaScript 对象，可以运行在任何平台上（如 React Native）
3. **状态管理**：将 UI 看作是状态的函数，`UI = f(state)`

### Virtual DOM 的结构

Virtual DOM 本质上是一个普通的 JavaScript 对象，它描述了 DOM 节点的结构：

```js
// 真实 DOM
// <div id="app">
//   <h1>Hello World</h1>
//   <p>这是一个段落</p>
// </div>

// 对应的 Virtual DOM
const virtualDOM = {
  type: 'div',
  props: {
    id: 'app',
    children: [
      {
        type: 'h1',
        props: {
          children: ['Hello World']
        }
      },
      {
        type: 'p',
        props: {
          children: ['这是一个段落']
        }
      }
    ]
  }
}
```

### Virtual DOM 的创建

在 React 中，JSX 语法会被 Babel 编译为 `React.createElement` 调用，最终生成 Virtual DOM 对象：

```jsx
// JSX 语法
const element = <h1 title="foo">Hello World</h1>

// 编译后
const element = React.createElement('h1', { title: 'foo' }, 'Hello World')

// 生成的 Virtual DOM 对象
const element = {
  type: 'h1',
  props: {
    title: 'foo',
    children: 'Hello World'
  }
}
```

## Diff 算法

### 为什么需要 Diff 算法

当组件的 state 或 props 发生变化时，React 会重新执行 render 函数，生成一棵新的 Virtual DOM 树。React 需要将这棵新树与旧的 Virtual DOM 树进行比较，找出差异，然后只更新发生变化的部分到真实 DOM 中。

如果采用完整的树对比算法，时间复杂度为 O(n³)，这对于大型应用来说是不可接受的。React 采用了一种启发式的 Diff 算法，将时间复杂度优化为 O(n)。

### Diff 算法的核心策略

React 的 Diff 算法基于以下三个假设：

#### 1. 两个不同类型的元素会产生不同的树

如果节点类型不同（如 `div` 变成 `span`，或组件 A 变成组件 B），React 会直接销毁旧树并创建新树，不会进行深度比较。

```jsx
// 渲染前
<div>
  <h1>Hello</h1>
  <p>World</p>
</div>

// 渲染后 - 根节点类型改变，整个子树都会被重建
<span>
  <h1>Hello</h1>
  <p>World</p>
</span>
```

#### 2. 开发者可以通过 key 属性暗示子节点在不同渲染之间保持稳定

对于列表渲染，React 使用 key 来识别哪些元素发生了变化、添加或删除了。

```jsx
// 没有 key 的情况 - 可能导致不必要的 DOM 操作
{items.map(item => <li>{item.name}</li>)}

// 使用 key 的情况 - React 可以准确识别变化
{items.map(item => <li key={item.id}>{item.name}</li>)}
```

#### 3. 同层比较

React 只会对同一层级的节点进行比较，不会跨层级比较。这意味着：

```jsx
// 渲染前
<div>
  <span>
    <p>Hello</p>
  </span>
</div>

// 渲染后 - React 不会将 div 下的 p 直接移动到 span 的位置
<div>
  <p>Hello</p>
</div>
```

### Diff 算法的执行过程

#### 1. Element Diff（同级元素比较）

对于同一层级的多个子节点，React 会通过 key 进行匹配：

```
旧列表: [A, B, C, D]
新列表: [A, C, B, D, E]

Diff 结果:
- A: 保持不变
- C: 移动
- B: 移动
- D: 保持不变
- E: 新增
```

如果没有 key，React 会按照索引顺序进行比较，这可能导致大量的不必要更新。

#### 2. Component Diff（组件比较）

- 同一类型的组件实例：保持实例不变，继续递归比较子节点
- 不同类型的组件：卸载旧组件，挂载新组件

#### 3. DOM Diff（属性比较）

对于同一类型的 DOM 节点，React 会比较它们的属性，只更新发生变化的属性：

```js
// 旧属性
{ className: 'active', id: 'app', style: { color: 'red' } }

// 新属性
{ className: 'active', id: 'app', style: { color: 'blue' } }

// 只更新 style.color 从 'red' 变为 'blue'
```

### key 的最佳实践

1. **使用唯一且稳定的 ID**：优先使用数据中的唯一标识符

```jsx
{users.map(user => (
  <UserCard key={user.id} user={user} />
))}
```

2. **避免使用索引作为 key**：当列表项会增删、排序时，使用索引会导致错误的复用

```jsx
// ❌ 不推荐
{items.map((item, index) => <Item key={index} {...item} />)}

// ✅ 推荐
{items.map(item => <Item key={item.uniqueId} {...item} />)}
```

3. **key 在兄弟节点中必须唯一**：key 只需要在当前层级唯一，不需要全局唯一

### Fiber 架构与 Diff

React 16 引入了 Fiber 架构，将 Diff 过程拆分为两个阶段：

#### Render 阶段（可中断）

- 执行 Diff 算法，计算需要更新的内容
- 这个过程是可中断的，可以暂停和恢复
- 不会产生副作用

#### Commit 阶段（不可中断）

- 将 Render 阶段的计算结果批量应用到真实 DOM
- 这个过程是不可中断的，必须一次性完成
- 会触发副作用（如生命周期钩子、ref 回调等）

### 性能优化技巧

1. **合理使用 key**：减少不必要的 DOM 操作
2. **使用 React.memo**：避免函数组件不必要的重新渲染
3. **使用 PureComponent**：避免类组件不必要的重新渲染
4. **使用 useMemo 和 useCallback**：缓存计算结果和回调函数
5. **拆分组件**：将大组件拆分为小组件，减少 Diff 范围
