## 一个中文占多少字节

一个中文字符占多少字节跟编码密切相关

- **UTF-8 编码**：一个中文字符通常占据 3 个字节

  UTF-8 是一种变长编码方式，对于英文字母和数字等 ASCII 字符，每个字符占用 1 个字节；而对于汉字等常用非 ASCII 字符，每个字符占用 3 个字节

- **GBK 编码**：一个中文字符通常占据 2 个字节

在 JavaScript 中，通常使用 UTF-16 编码，而网页通常是 UTF-8 编码，所以使用 JavaScript 内部的字符串方法时，实际上内部会做一个转换，是以 UTF-16 为准

```js
"😂".length; // 2
```

码元（code unit），是指存储字符的最小单位，这里即 2 个字节。所以 length 就是码元的个数，上面的表情符号用了 4 个字节，所以是 2 个码元，所以 length 是 2

**与前端的关系**：
- `Content-Type: text/html; charset=utf-8` 决定了浏览器如何解码服务端返回的字节流
- HTTP 传输、localStorage、IndexedDB 等存储大小计算都依赖底层编码
- `TextEncoder` / `TextDecoder` API 可在 JS 中直接操作字节流

## 0.1 + 0.2 !== 0.3

> JavaScript 中的 Number 类型有 18437736874454810627（即 `2^64 - 2^53 + 3`）个值。9007199254740991（16 位数字，即 `2^53 - 1`）就是 JavaScript 能安全表示的最大整数（`Number.MAX_SAFE_INTEGER`）

在 IEEE 754 标准中，浮点数的表示是有限的，而 0.1 和 0.2 在二进制下是无限循环小数

- 十进制小数转为二进制，采用的是"乘 2 取整，顺序排列"法，0.1 在由十进制转换为二进制时出现了无限循环，所以无法精准转换，这也就是 `0.1 + 0.2 !== 0.3`
- 但是 `0.1 + 0.1 === 0.2`，为什么呢？

在 IEEE 754 标准中的 64 位浮点数的小数部分，最多有 53 位，2 的 53 次方就是 16 位数字，所以小数部分最多展示 16 位

- 0.1 + 0.1 正好 16 位（四舍五入）截断，等于 0.2

```js
(0.1).toPrecision(16); // "0.1000000000000000"
(0.1).toPrecision(17); // "0.10000000000000001"
```

如何避免？

- 使用将数字转化为字符串，然后模拟加法运算，一些库就是这样实现的，所以更建议使用成熟的第三方库
- 当然，这种模拟运算性能并不好，相比于支持 decimal 类型的语言

## 内存管理

### 栈 vs 堆

| 特性 | 栈（Stack） | 堆（Heap） |
| --- | --- | --- |
| 分配方式 | 自动分配/释放 | 手动或 GC 管理 |
| 存储内容 | 基本类型值、函数调用帧、局部变量 | 对象、数组等引用类型 |
| 访问速度 | 快 | 相对慢 |
| 内存空间 | 较小 | 较大 |
| 连续性 | 连续内存 | 非连续内存 |

**栈**：后进先出（LIFO），函数调用时压栈，函数返回时弹栈。存储基本类型（`number`、`string`、`boolean`、`null`、`undefined`、`symbol`、`bigint`）

**堆**：存储引用类型（`Object`、`Array`、`Function` 等），对象在堆中分配内存，变量在栈中保存指向堆中对象的引用地址

```js
let a = 1; // 基本类型，值直接存储在栈上
let b = { name: 'test' }; // 引用类型，对象存储在堆上，b 在栈上保存指向堆的引用
```

### 前端中的内存分配

**基本类型**：值直接存储在栈上，按值传递

```js
let a = 10;
let b = a; // 复制的是值
b = 20;
console.log(a); // 10，不受影响
```

**引用类型**：对象存储在堆上，变量保存的是内存地址，按引用传递

```js
let obj1 = { count: 1 };
let obj2 = obj1; // 复制的是引用地址
obj2.count = 2;
console.log(obj1.count); // 2，两个变量指向同一个对象
```

### 内存泄漏

**常见场景**：

1. **意外全局变量**：未声明的变量会自动挂载到 `window` 上

```js
function leak() {
  lostVar = 'I am lost'; // 未用 let/const/var 声明，成为全局变量
}
```

2. **未清理的定时器**

```js
let timer = setInterval(() => {
  console.log('running');
});
// 忘记 clearInterval(timer)，回调函数及其闭包引用的内存无法回收
```

3. **闭包引用**：闭包持有外部函数变量的引用，导致变量无法被回收

```js
function createLeak() {
  let largeData = new Array(1000000).fill('data');
  return function () {
    return largeData; // 闭包持有 largeData 引用
  };
}
```

4. **未移除的事件监听器**

```js
let btn = document.getElementById('btn');
btn.addEventListener('click', handler);
// 元素从 DOM 移除但未 removeEventListener，handler 和 btn 引用无法回收
```

5. **Detached DOM**：从 DOM 树移除的元素仍被 JavaScript 引用

**检测手段**：

- Chrome DevTools → Memory → Heap Snapshot / Allocation instrumentation
- `performance.memory` API（Chrome 专用）
- 使用 `console.table()` 观察对象引用

### 垃圾回收机制

JavaScript 使用自动垃圾回收（GC），主流策略是**标记-清除**（Mark-and-Sweep）

**V8 引擎的分代回收机制**：

V8 将堆内存分为两个区域：

1. **新生代**（Young Generation）：存放新创建的对象，大小约 1-2MB
   - 使用 **Scavenge 算法**（复制算法）
   - 新生代再分为两个等大的空间：From 和 To
   - GC 时，将 From 空间中存活的对象复制到 To 空间，然后清空 From
   - 对象经历一次 GC 后仍存活，复制到 To 空间时年龄 +1
   - 年龄达到阈值（通常 8 次），晋升到老生代

2. **老生代**（Old Generation）：存放长期存活的对象
   - 使用 **标记-清除**（Mark-Sweep） + **标记-整理**（Mark-Compact）
   - 标记阶段：从根对象（全局对象、执行上下文等）出发，标记所有可达对象
   - 清除阶段：回收未被标记的对象
   - 整理阶段：将存活对象向一端移动，消除内存碎片

```js
// 新生代对象
let temp = new Array(10); // 短期存在，GC 后很快回收

// 老生代对象
let cache = new Map(); // 长期存在，多次 GC 后晋升到老生代
```

## 浏览器内核

### 浏览器架构与线程模型

浏览器是多线程的，主要包含以下线程：

| 线程 | 职责 |
| --- | --- |
| **GUI 渲染线程** | 解析 HTML/CSS，构建 DOM/CSSOM，布局、绘制页面。与 JS 引擎线程互斥 |
| **JS 引擎线程** | 解析和执行 JavaScript 代码。与渲染线程互斥，避免渲染不一致 |
| **事件触发线程** | 管理事件队列，当事件满足触发条件时放入任务队列，等待 JS 线程执行 |
| **定时触发线程** | 管理 `setTimeout`/`setInterval` 计时，时间到达后将任务放入事件队列 |
| **异步 HTTP 请求线程** | 处理 `XMLHttpRequest`/`Fetch` 等异步请求，状态改变时回调放入事件队列 |
| **主线程** | 协调以上各线程的工作 |

**关键点**：GUI 渲染线程和 JS 引擎线程是互斥的。JS 执行期间，渲染线程会被挂起，JS 执行完毕后才恢复渲染。这就是为什么长时间运行的 JS 会阻塞页面渲染

```js
// 长时间运行的 JS 会阻塞渲染
let start = Date.now();
while (Date.now() - start < 3000) {
  // 这 3 秒内页面无法响应、无法渲染
}
```

### 浏览器渲染流程

```
HTML 解析 → DOM Tree
CSS 解析 → CSSOM Tree
              ↓
        Render Tree（渲染树）
              ↓
         Layout（布局/重排）
              ↓
         Paint（绘制/重绘）
              ↓
       Composite（合成）
```

1. **HTML 解析**：HTML 解析器将 HTML 文本解析为 Token，再构建 DOM Tree
2. **CSS 解析**：CSS 解析器将 CSS 文本解析为 CSS Rule Node，构建 CSSOM Tree
3. **Render Tree 构建**：合并 DOM Tree 和 CSSOM Tree，跳过 `display: none` 和 `<head>` 中的元素，生成 Render Tree
4. **Layout（重排）**：计算每个节点在视口中的大小和位置
5. **Paint（重绘）**：将 Render Tree 的各个节点绘制到屏幕上，填充颜色、文字、图片等
6. **Composite（合成）**：将页面分成多个图层，由 GPU 合成最终画面

**优化原则**：避免触发 Layout，尽量只触发 Paint，最好只触发 Composite

## 网络基础

### TCP 三次握手

```
客户端                          服务端
  | -- SYN(seq=x) ------------> |   第一次：客户端发送 SYN，进入 SYN_SENT 状态
  | <--- SYN+ACK(seq=y,ack=x+1)-|   第二次：服务端回复 SYN+ACK，进入 SYN_RCVD 状态
  | -- ACK(ack=y+1) ----------> |   第三次：客户端发送 ACK，进入 ESTABLISHED 状态
  | <--- ACK ------------------ |   服务端进入 ESTABLISHED 状态
```

**为什么是三次？**
- 两次握手无法防止已失效的连接请求到达服务端，导致资源浪费
- 三次握手可以确保双方的发送和接收能力都正常

**为什么不能用两次？**
- 如果只有两次，客户端发出的延迟报文可能到达服务端，服务端会创建连接但客户端并不知道，导致服务端资源被占用

### TCP 四次挥手

```
客户端                          服务端
  | -- FIN(seq=u) ------------> |   第一次：客户端发送 FIN，进入 FIN_WAIT_1
  | <--- ACK(ack=u+1) -------->|   第二次：服务端回复 ACK，进入 CLOSE_WAIT
  | <--- FIN(seq=v) ---------->|   第三次：服务端发送 FIN，进入 LAST_ACK
  | -- ACK(ack=v+1) ----------> |   第四次：客户端回复 ACK，进入 TIME_WAIT
  |                             |   经过 2MSL 后，客户端关闭连接
```

**为什么是四次？**
- TCP 是全双工通信，每个方向都需要独立关闭
- 服务端收到 FIN 时可能还有数据要发送，所以 ACK 和 FIN 分开

**TIME_WAIT 状态的作用**：
- 确保最后一个 ACK 能够到达服务端
- 防止已失效的连接报文段出现在新连接中

### TCP vs UDP

| 特性 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠，保证数据到达和顺序 | 不可靠，不保证到达 |
| 速度 | 较慢（握手、确认、重传） | 快 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有 | 无 |
| 应用场景 | HTTP、HTTPS、FTP | 视频直播、语音通话、DNS、QUIC |

### 拥塞控制

TCP 拥塞控制通过四个算法协同工作：

1. **慢启动**（Slow Start）
   - 初始拥塞窗口（cwnd）为 1 个 MSS
   - 每经过一个 RTT，cwnd 翻倍（指数增长）
   - 当 cwnd 达到慢启动阈值（ssthresh）时，进入拥塞避免阶段

2. **拥塞避免**（Congestion Avoidance）
   - cwnd 每经过一个 RTT 增加 1 个 MSS（线性增长）
   - 防止窗口增长过快导致拥塞

3. **快重传**（Fast Retransmit）
   - 接收方发现丢包后，重复发送已收到数据的 ACK
   - 发送方收到 3 个重复 ACK，立即重传丢失的数据包，无需等待超时

4. **快恢复**（Fast Recovery）
   - 收到 3 个重复 ACK 后，将 ssthresh 设为当前 cwnd 的一半
   - cwnd 设为 ssthresh + 3，然后进入拥塞避免阶段
   - 不需要重新慢启动

### TLS/SSL 握手过程

以 TLS 1.3 为例：

```
客户端                          服务端
  | -- ClientHello -----------> |   客户端发送支持的加密套件、随机数等
  | <--- ServerHello ----------|   服务端选择加密套件、发送随机数
  | <--- Certificate --------->|   服务端发送证书（含公钥）
  | <--- EncryptedExtensions ->|   加密的扩展信息
  | <--- Finished ------------>|   加密的握手完成消息
  | -- KeyUpdate ------------->|   密钥更新
  | -- Finished --------------> |   客户端确认
  |                             |   之后所有通信使用对称加密
```

**核心流程**：
1. 客户端发送 `ClientHello`，包含支持的 TLS 版本、加密套件列表、客户端随机数
2. 服务端回复 `ServerHello`，选定加密套件、发送服务端随机数
3. 服务端发送证书，客户端验证证书合法性（CA 签名、有效期、域名匹配）
4. 双方基于随机数和密钥交换算法（如 ECDHE）生成共享密钥
5. 后续通信使用对称加密（如 AES-GCM），保证效率和安全性

**为什么 HTTPS 比 HTTP 慢？**
- 多了一次 TCP 连接（TLS 握手在 TCP 之上）
- 非对称加密计算开销大
- 优化手段：TLS Session Resumption、TLS 1.3 减少握手往返次数（1-RTT 甚至 0-RTT）

## 操作系统基础

### 进程 vs 线程

| 特性 | 进程（Process） | 线程（Thread） |
| --- | --- | --- |
| 定义 | 操作系统资源分配的基本单位 | CPU 调度的基本单位 |
| 内存空间 | 独立地址空间 | 共享进程的地址空间 |
| 通信方式 | IPC（进程间通信） | 共享内存、锁、信号量 |
| 创建开销 | 大 | 小 |
| 切换开销 | 大 | 小 |
| 稳定性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程崩溃 |

**进程间通信方式**：
- 管道（Pipe）
- 消息队列（Message Queue）
- 共享内存（Shared Memory）
- 信号量（Semaphore）
- Socket

**前端中的体现**：
- 浏览器每个标签页是一个独立进程
- 浏览器主进程、渲染进程、GPU 进程、网络进程等分工协作
- Web Worker 运行在独立线程中，不阻塞主线程

```js
// Web Worker 示例：在独立线程中执行计算密集型任务
// worker.js
self.onmessage = function (e) {
  let result = heavyComputation(e.data);
  self.postMessage(result);
};

// main.js
let worker = new Worker('worker.js');
worker.postMessage(largeData);
worker.onmessage = function (e) {
  console.log('计算结果:', e.data);
};
```

### 协程

**概念**：协程是用户态的轻量级线程，由程序自身控制调度，而非操作系统

**协程 vs 线程**：

| 特性 | 线程 | 协程 |
| --- | --- | --- |
| 调度 | 操作系统内核调度 | 用户态程序调度 |
| 切换开销 | 大（涉及上下文切换） | 极小 |
| 内存占用 | 大（通常 1-2MB 栈空间） | 小（几 KB） |
| 并发模型 | 抢占式 | 协作式（主动让出控制权） |

**前端中的体现**：
- JavaScript 的 `async/await` 本质上是协程的思想
- `await` 表达式主动让出执行权，等待 Promise 完成后恢复执行
- 单线程模型下通过事件循环 + 协程实现高并发

```js
async function fetchData() {
  // await 让出控制权，不阻塞其他任务
  let res1 = await fetch('/api/data1');
  let res2 = await fetch('/api/data2');
  return { res1, res2 };
}
```

### 调度算法

1. **时间片轮转**（Round Robin）
   - 每个进程分配一个固定时间片
   - 时间片用完，强制切换到下一个就绪进程
   - 公平性好，但上下文切换开销大
   - 浏览器渲染中，60fps 对应每帧约 16.6ms 的时间片

2. **优先级调度**
   - 优先级高的进程优先获得 CPU
   - 可分静态优先级和动态优先级
   - 浏览器中，用户交互事件的优先级高于动画帧，动画帧优先级高于普通任务

```js
// 浏览器任务优先级（从高到低）
// 1. 用户输入事件（click, keydown 等）
// 2. requestAnimationFrame（动画帧）
// 3. requestIdleCallback（空闲任务）
// 4. setTimeout/setInterval（定时器）
// 5. Promise.then / Microtask
```

## 数据编码

### 字符编码

| 编码 | 特点 | 字节范围 |
| --- | --- | --- |
| **ASCII** | 最早的字符编码，128 个字符 | 1 字节 |
| **UTF-8** | 变长编码，1-4 字节，ASCII 兼容 | 1-4 字节 |
| **UTF-16** | 变长编码，2 或 4 字节，JS 内部使用 | 2-4 字节 |
| **UTF-32** | 定长编码，每个字符 4 字节 | 4 字节 |

**UTF-8 编码规则**：
- 1 字节：`0xxxxxxx`（ASCII 字符，0-127）
- 2 字节：`110xxxxx 10xxxxxx`（128-2047）
- 3 字节：`1110xxxx 10xxxxxx 10xxxxxx`（2048-65535，大部分汉字）
- 4 字节：`11110xxx 10xxxxxx 10xxxxxx 10xxxxxx`（65536 以上）

**UTF-16 编码规则**：
- BMP（基本多文种平面，U+0000 ~ U+FFFF）：2 字节
- 辅助平面（U+10000 ~ U+10FFFF）：4 字节，使用代理对（Surrogate Pair）

```js
// BMP 字符：2 字节
'中'.codePointAt(0); // 20013

// 辅助平面字符：代理对，4 字节
'😂'.codePointAt(0); // 128514
'😂'.charCodeAt(0); // 55357（高代理项）
'😂'.charCodeAt(1); // 56834（低代理项）
```

### Base64 编码原理

Base64 使用 64 个可打印字符（`A-Z`、`a-z`、`0-9`、`+`、`/`）对二进制数据进行编码

**编码规则**：
1. 将每 3 个字节（24 bit）分为 4 组，每组 6 bit
2. 每组 6 bit 作为索引，从 Base64 字符表中查找对应字符
3. 如果最后剩余 1 个字节，补 2 个 `=`；剩余 2 个字节，补 1 个 `=`

```
3 字节 (24bit) → 4 个 Base64 字符
输入: [ABC] → 输出: 4 个字符
编码后体积增加约 33%
```

```js
// 前端中的 Base64 使用
let str = 'Hello World';
let encoded = btoa(str); // SGVsbG8gV29ybGQ
let decoded = atob(encoded); // Hello World

// 图片转 Base64，减少 HTTP 请求
// data:image/png;base64,iVBORw0KGgoAAAANSUhEUg...
```

**应用场景**：
- 小图片 inline 嵌入（减少 HTTP 请求）
- 数据 URI 方案
- 传输二进制数据的文本安全编码

### URL 编码

URL 编码（Percent-Encoding）将特殊字符转换为 `%` + 两位十六进制数

**需要编码的字符**：空格、中文、`#`、`&`、`=`、`?`、`/` 等保留字符

```js
// 编码
encodeURIComponent('你好 world'); // "%E4%BD%A0%E5%A5%BD%20world"
encodeURI('https://example.com/path?name=你好'); // 保留 URL 结构字符

// 解码
decodeURIComponent("%E4%BD%A0%E5%A5%BD%20world"); // "你好 world"

// 区别
encodeURI('a=b&c=d'); // "a=b&c=d"（不编码 & = ?）
encodeURIComponent('a=b&c=d'); // "a%3Db%26c%3Dd"（全部编码）
```

**`encodeURI` vs `encodeURIComponent`**：
- `encodeURI`：编码整个 URL，保留 `: / ? & =` 等 URL 结构字符
- `encodeURIComponent`：编码 URL 参数值，所有非安全字符都编码

## 前端相关的 GPU 知识

### 合成层（Composite Layer）

**什么是合成层？**
浏览器将页面拆分为多个图层，每个图层独立绘制和合成。GPU 负责将多个图层合成为最终画面

**如何触发合成层？**
以下情况会促使浏览器创建独立的合成层：
- 使用 `transform` 或 `opacity` 动画
- 使用 `will-change` 声明
- 使用 `backface-visibility: hidden`
- `<video>`、`<canvas>`、`<iframe>` 元素
- 使用 `position: fixed`
- 使用 `filter`、`clip-path`、`mix-blend-mode` 等 CSS 属性

```js
// 强制元素提升为合成层
.element {
  will-change: transform;
  // 或
  transform: translateZ(0);
  // 或
  backface-visibility: hidden;
}
```

**合成层优化原则**：
- 不要滥用合成层，过多的合成层会消耗大量内存
- 只对有动画效果的元素创建合成层
- 动画结束后移除 `will-change`，释放资源

### 浏览器渲染管线

```
Style（样式计算） → Layout（布局） → Paint（绘制） → Composite（合成）
```

| 阶段 | 说明 | 触发条件 |
| --- | --- | --- |
| **Style** | 计算每个元素最终的 CSS 样式 | CSS 属性变化 |
| **Layout** | 计算元素的大小和位置 | 尺寸、位置、内容变化 |
| **Paint** | 将元素绘制到像素 | 颜色、边框、阴影等变化 |
| **Composite** | 将图层合成为最终画面 | 图层变换、透明度变化 |

**性能影响**：Layout > Paint > Composite

**各阶段可优化的属性**：
- 只触发 Composite：`transform`、`opacity`
- 只触发 Paint：`color`、`background-color`、`border-color`、`box-shadow`
- 触发 Layout：`width`、`height`、`margin`、`padding`、`top`、`left`、`font-size`

### 动画性能优化

**核心原则**：只在 Composite 阶段操作，避免触发 Layout 和 Paint

**推荐使用的属性**：
- `transform`：位移、旋转、缩放
- `opacity`：透明度

```js
// ❌ 低性能：触发 Layout + Paint
.element {
  left: 100px;
  top: 50px;
}

// ✅ 高性能：只触发 Composite
.element {
  transform: translate(100px, 50px);
}

// ❌ 低性能：触发 Paint
.element {
  color: red;
}

// ✅ 高性能：只触发 Composite
.element {
  opacity: 0.5;
}
```

**其他优化手段**：
- 使用 `requestAnimationFrame` 替代 `setTimeout` 做动画
- 使用 CSS `animation` / `transition` 代替 JS 动画（浏览器可以优化）
- 使用 `will-change` 提前告知浏览器需要优化的属性
- 减少重排和重绘：批量修改 DOM，使用 `DocumentFragment`
- 使用 `transform: translate3d()` 强制 GPU 加速

```js
// 使用 requestAnimationFrame 做高性能动画
function animate() {
  element.style.transform = `translateX(${position}px)`;
  position += speed;
  requestAnimationFrame(animate);
}
requestAnimationFrame(animate);
```
