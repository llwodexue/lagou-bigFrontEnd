# 前端性能优化

## 一、网络与资源加载

#### 1. DNS Prefetch

浏览器解析 HTML 中的外链资源时，若域名与当前页面不同，需额外进行 DNS 解析。通过 `<link rel="dns-prefetch">` 可提前解析并缓存 DNS，减少后续请求延迟。

```html
<link rel="dns-prefetch" href="//img.alicdn.com">
```

#### 2. 资源加载hints 对比

| hint | 作用 | 适用场景 |
|------|------|----------|
| `dns-prefetch` | 仅 DNS 解析 | 跨域资源 |
| `preconnect` | DNS + TCP + TLS | 已知需要连接的第三方域名 |
| `prefetch` | 预下载资源（低优先级） | 下一页可能用到的资源 |
| `preload` | 预加载关键资源（高优先级） | 当前页必需的字体/脚本/图片 |

```html
<link rel="preconnect" href="https://fonts.googleapis.com" crossorigin>
<link rel="preload" href="critical.woff2" as="font" type="font/woff2" crossorigin>
<link rel="prefetch" href="next-page.js">
```

#### 3. async vs defer

两者都实现脚本异步下载，不阻塞 HTML 解析，但执行时机不同：

```html
<!-- 下载完毕立即执行，无序 -->
<script src="a.js" async></script>
<!-- DOMContentLoaded 后按顺序执行 -->
<script src="a.js" defer></script>
```

- `async`：下载完成即执行，多个 async 脚本执行顺序不确定，适合独立脚本
- `defer`：HTML 解析完成后按文档顺序执行，适合有依赖关系的脚本

#### 4. 域名收敛

外链图片域名分散导致 DNS 解析开销大。统一收敛到 CDN 域名，配合 dns-prefetch 效果更佳：

```
aaa.com/jjj.png -> mycdn.com/aaa/jjj.png
bbb.com/kkk.png -> mycdn.com/bbb/kkk.png
```

#### 5. 图片优化

- **现代格式**：优先使用 WebP/AVIF，比 JPEG 小 25-35%
- **响应式图片**：使用 `srcset` 按设备加载合适尺寸
- **懒加载**：`loading="lazy"` 延迟加载视口外图片
- **CDN 裁剪**：通过 URL 参数动态裁剪尺寸

```html
<img
  src="photo.jpg"
  srcset="photo-400w.jpg 400w, photo-800w.jpg 800w, photo-1200w.jpg 1200w"
  sizes="(max-width: 600px) 400px, 800px"
  loading="lazy"
  alt="description"
/>
```

```text
<!-- CDN 裁剪示例 -->
http://cdn.example.com/img.jpg?x-oss-process=image/crop,x_100,y_50,w_400,h_300
```

> - 不透明图片用 jpg，带透明用 png/WebP
> - 动效优先用 APNG 或 Lottie，避免 GIF
> - 仅极小图标考虑 base64 内联

#### 6. 字体优化

```css
@font-face {
  font-family: 'CustomFont';
  src: url('font.woff2') format('woff2');
  font-display: swap; /* 避免 FOIT，先显示后备字体 */
}
```

```html
<!-- 关键字体 preload -->
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
```

- `font-display: swap`：先渲染后备字体，自定义字体加载完成后替换
- `font-display: optional`：网络慢时直接使用后备字体，不替换

#### 7. 资源 Combo

HTTP/1.x 时代，多个小文件会产生大量 TCP 握手开销。服务端将多个资源合并为一个请求返回：

```
xx.com/a.js + xx.com/b.js + xx.com/c.js
-> xx.com/combo?a.js,b.js,c.js
```

> HTTP/2 多路复用后，combo 收益降低，需权衡缓存命中率。

#### 8. 在线 Polyfill

按需下发兼容性垫片，避免新浏览器加载无用代码：

```html
<!-- polyfill.io 根据浏览器能力自动返回所需 polyfill -->
<script src="https://polyfill.io/v3/polyfill.min.js?features=Promise,Object.assign"></script>
```

另一种方案：在加载主 bundle 前检测 ES 特性支持情况，分别加载 modern/legacy bundle。

#### 9. 分离静态资源域名

静态资源请求无需携带 Cookie，使用独立域名可避免 Cookie 上传，减少请求体积。

#### 10. GZIP & Brotli 压缩

- **GZIP**：基于 Deflate，广泛支持，压缩比一般
- **Brotli**：Google 开发，压缩比比 GZIP 高 15-20%，现代浏览器均支持

```nginx
# Nginx 配置示例
gzip on;
gzip_types text/css application/javascript;

brotli on;
brotli_types text/css application/javascript;
```

> 压缩率并非越高越好，过高压缩会消耗服务端 CPU。

#### 11. 避免 "闪动 Loading"

动态页面数据返回极快时，Loading/骨架屏会出现瞬间闪动。优化方案：延迟 200ms+ 再展示占位内容。

```js
// 延迟展示 loading
const timer = setTimeout(() => {
  setLoading(true);
}, 200);

fetchData().then(() => {
  clearTimeout(timer);
  setLoading(false);
  setData(res);
});
```

#### 12. 使用新语法

V8 等引擎对新语法有专门优化。策略：modern 浏览器用新语法，legacy 浏览器用转译代码。

#### 13. LocalStorage / IndexedDB 缓存

数据更新不频繁的场景（如个人博客），可用 LocalStorage 缓存 HTML，页面进入时先渲染缓存，接口返回后再 Diff 更新。大体积数据优先使用 IndexedDB。

#### 14. 组件级缓存

组件代码单独打包并缓存到 LocalStorage（带版本号），多页面共享同一份组件缓存，加速首次访问。

#### 15. Service Worker

Service Worker 作为浏览器与服务端之间的代理层，可实现离线缓存、请求拦截、消息推送等能力，是 PWA 的核心技术。

#### 16. Web Worker

将密集计算或耗时任务放到 Worker 线程，避免阻塞主线程：

```js
// main.js
const worker = new Worker('compute.js');
worker.postMessage({ data: largeArray });
worker.onmessage = (e) => console.log('结果:', e.data);

// compute.js
self.onmessage = (e) => {
  const result = heavyComputation(e.data.data);
  self.postMessage(result);
};
```

## 二、Core Web Vitals

#### LCP (Largest Contentful Paint)

最大内容绘制时间，衡量页面主要内容加载速度。**目标 < 2.5s**。

**优化方案**：
- 优化/压缩最大元素（通常是图片或英雄横幅）
- 使用 `preload` 预加载关键资源
- 服务端渲染 (SSR) 或静态生成
- 避免使用大型 CSS 阻塞渲染

```html
<link rel="preload" as="image" href="hero.webp">
```

#### INP (Interaction to Next Paint)

交互到下次绘制时间，替代 FID，衡量整体交互响应性。**目标 < 200ms**。

INP 取所有交互中的最差值（p98），而非仅首次。

**优化方案**：
- 拆分长任务，使用 `setTimeout` / `scheduler.postTask` 分片
- 减少事件处理器中的同步工作
- 使用 Web Worker 处理复杂计算
- 优化 CSS 布局，减少重排

#### CLS (Cumulative Layout Shift)

累积布局偏移，衡量视觉稳定性。**目标 < 0.1**。

**优化方案**：
- 图片/视频/iframe 设置明确的 `width`/`height` 或 `aspect-ratio`
- 动态插入内容时放在已有内容之前
- 字体加载使用 `font-display: optional`
- 避免使用 `fixed` 定位的弹窗突然遮挡内容

```html
<!-- 为图片预留空间 -->
<img src="photo.jpg" style="aspect-ratio: 16/9;" alt="photo">
```

#### TTFB (Time to First Byte)

首字节时间，从请求发起到接收第一个字节的时间。**目标 < 0.8s**。

**优化方案**：
- 使用 CDN 加速
- 优化服务端逻辑和数据库查询
- 启用 HTTP 缓存
- 考虑 Edge Computing / 边缘渲染

#### 测量 Core Web Vitals

```js
// 使用 web-vitals 库
import { onLCP, onINP, onCLS } from 'web-vitals';

onLCP((metric) => {
  console.log('LCP:', metric.value, 'ms');
  sendToAnalytics(metric);
});

onINP((metric) => {
  console.log('INP:', metric.value, 'ms');
  sendToAnalytics(metric);
});

onCLS((metric) => {
  console.log('CLS:', metric.value);
  sendToAnalytics(metric);
});

function sendToAnalytics(metric) {
  navigator.sendBeacon('/analytics', JSON.stringify(metric));
}
```

## 三、浏览器渲染优化

#### 重排 (Reflow) vs 重绘 (Repaint)

- **重排**：DOM 结构或元素几何属性（宽高、位置）变化时，浏览器重新计算布局。开销大。
- **重绘**：仅视觉样式变化（颜色、背景），不影响布局。开销较小。

```js
// 触发重排的操作
element.offsetWidth;       // 读取布局属性
element.style.width = '100px';
element.appendChild(node);

// 仅触发重绘的操作
element.style.color = 'red';
element.style.backgroundColor = '#fff';
```

#### 避免重排

**1. DOM 读写分离**：批量读取后再批量写入，避免读写交替触发多次重排。

```js
// ❌ 差：每次写入后紧跟读取，触发多次重排
const w = el.offsetWidth;
el.style.width = w + 10 + 'px';
const h = el.offsetHeight;
el.style.height = h + 10 + 'px';

// ✅ 好：先读后写，只触发一次重排
const w = el.offsetWidth;
const h = el.offsetHeight;
el.style.width = w + 10 + 'px';
el.style.height = h + 10 + 'px';
```

**2. 使用 DocumentFragment**：批量操作 DOM 后再一次性插入。

```js
const fragment = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  fragment.appendChild(li);
}
ul.appendChild(fragment); // 只触发一次重排
```

**3. 离线样式修改**：脱离文档流修改，完成后重新插入。

```js
el.style.display = 'none'; // 脱离渲染
// ... 多次样式修改 ...
el.style.display = '';     // 恢复，只触发一次重排
```

#### 强制同步布局 (Forced Synchronous Layout)

在写入样式后、浏览器刷新前立即读取布局属性，会强制浏览器提前完成布局计算，造成性能损耗。

```js
// ❌ 强制同步布局
el.style.width = '500px';
console.log(el.offsetWidth); // 读取布局属性，强制浏览器立即重排

// ✅ 使用 requestAnimationFrame 推迟读取
el.style.width = '500px';
requestAnimationFrame(() => {
  console.log(el.offsetWidth); // 在浏览器渲染帧前读取，避免强制同步
});
```

#### 合成层优化

将元素提升为独立合成层，可避免触发重排重绘，由 GPU 直接合成。

```css
/* 触发合成层的属性 */
.promoted {
  will-change: transform;
  /* 或使用以下属性自动触发 */
  transform: translateZ(0);
  /* opacity 变化也走合成层 */
}
```

- `will-change`：提前告知浏览器哪些属性会变化，提前创建合成层
- `transform` / `opacity`：变化只触发合成，不触发重排重绘
- 注意：滥用合成层会消耗内存，仅在动画元素上使用

#### CSS Containment — 布局隔离

CSS Containment 允许开发者显式声明元素的隔离级别，限制重排/重绘的影响范围。

```css
/* 完全隔离：布局、样式、绘制、尺寸均独立 */
.contained {
  contain: strict; /* 等价于 layout + style + paint + size */
}

/* 仅布局隔离：元素内部布局变化不影响外部 */
.layout-isolated {
  contain: layout;
}

/* 仅绘制隔离：元素不可见时（如 overflow:hidden），跳过绘制 */
.paint-isolated {
  contain: paint;
  overflow: hidden;
}

/* 组合使用 */
.widget {
  contain: layout paint;
}
```

**适用场景：**
- 虚拟列表中的独立组件
- 第三方 widget（如广告、评论组件）
- 大型应用中模块化的子应用
- 与 Web Components 配合使用，实现真正的样式/布局隔离

**contain-intrinsic-size — 预留尺寸：**

```css
.lazy-load {
  contain: intrinsic-size;
  contain-intrinsic-size: 0 300px; /* 宽度自适应，高度预留 300px */
}
/* 元素未渲染时仍占位，避免 CLS（布局偏移） */
```

## 四、JavaScript 性能

#### 长任务检测与拆分

长任务（> 50ms）会阻塞主线程，导致页面无法响应用户交互。

```js
// 检测长任务
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 50) {
      console.warn('长任务:', entry.name, entry.duration, 'ms');
    }
  }
});
observer.observe({ entryTypes: ['longtask'] });

// 拆分长任务
function processInChunks(array, chunkSize = 100) {
  let index = 0;

  function processChunk() {
    const end = Math.min(index + chunkSize, array.length);
    for (; index < end; index++) {
      processItem(array[index]);
    }
    if (index < array.length) {
      setTimeout(processChunk, 0); // 让出主线程
    }
  }
  processChunk();
}
```

#### requestAnimationFrame vs requestIdleCallback

```js
// requestAnimationFrame：每帧执行，适合动画
requestAnimationFrame(() => {
  updateAnimation();
});

// requestIdleCallback：空闲时执行，适合低优先级任务
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0) {
    doLowPriorityWork();
  }
});
```

- `requestAnimationFrame`：在浏览器渲染前调用，保证与刷新率同步（~60fps）
- `requestIdleCallback`：在帧空闲时调用，不阻塞用户交互

#### 内存泄漏检测

常见泄漏场景：
- 未清理的定时器/事件监听
- 闭包持有大对象引用
- 脱离 DOM 但仍被引用的元素

```js
// Chrome DevTools Memory 面板操作步骤：
// 1. 打开 Memory 面板
// 2. 拍摄 Heap Snapshot
// 3. 执行可能泄漏的操作
// 4. 再次拍摄 Snapshot
// 5. 对比两个 Snapshot，查找未释放的对象

// 检测泄漏的定时器
const timers = [];
const originalSetInterval = window.setInterval;
window.setInterval = function (fn, delay) {
  const id = originalSetInterval(fn, delay);
  timers.push(id);
  return id;
};
console.log('活跃定时器:', timers.length);
```

## 五、缓存策略

#### 浏览器缓存

**强缓存**：浏览器直接使用本地缓存，不发请求。
- `Cache-Control: max-age=31536000`
- `Expires: <absolute-time>`（已过时，优先用 Cache-Control）

**协商缓存**：浏览器发请求询问资源是否更新。
- `ETag` / `If-None-Match`：基于内容哈希，精确但服务端开销大
- `Last-Modified` / `If-Modified-Since`：基于修改时间，简单但不够精确

```
# 强缓存命中 -> 直接使用缓存，状态码 200 (from disk/memory cache)
# 协商缓存命中 -> 返回 304 Not Modified，浏览器使用本地缓存
# 协商缓存未命中 -> 返回 200 + 新资源
```

#### HTTP 缓存头配置

```nginx
# 静态资源（带 hash 文件名）：强缓存，长期有效
location ~* \.(js|css|png|jpg|woff2)$ {
  expires 1y;
  add_header Cache-Control "public, immutable";
}

# HTML：不缓存或短缓存
location ~* \.html$ {
  add_header Cache-Control "no-cache";
  add_header ETag "";
}
```

#### Service Worker 缓存策略

```js
// Cache First：缓存优先，离线友好（静态资源）
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.open('v1').then((cache) =>
      cache.match(event.request).then((cached) =>
        cached || fetch(event.request).then((response) => {
          return cache.put(event.request, response.clone()).then(() => response);
        })
      )
    )
  );
});

// Network First：网络优先，缓存兜底（API 数据）
self.addEventListener('fetch', (event) => {
  event.respondWith(
    fetch(event.request)
      .then((response) => {
        const clone = response.clone();
        caches.open('v1').then((cache) => cache.put(event.request, clone));
        return response;
      })
      .catch(() => caches.match(event.request))
  );
});

// Stale While Revalidate：先返缓存，后台更新（兼顾速度和新鲜度）
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.open('v1').then((cache) =>
      cache.match(event.request).then((cached) => {
        const fetchPromise = fetch(event.request).then((response) => {
          cache.put(event.request, response.clone());
          return response;
        });
        return cached || fetchPromise;
      })
    )
  );
});
```

#### CDN 缓存策略

- 静态资源：文件名带 hash，设置长期缓存（1年）+ `immutable`
- HTML：短缓存或 `no-cache`，配合协商缓存
- API 接口：按需设置 `max-age`，或使用 `no-cache` 强制协商

## 六、现代优化手段

#### HTTP/2 Server Push

服务端主动推送资源到客户端，减少 RTT。但需权衡：可能推送客户端已缓存的资源。

```nginx
http2 on;
# 注意：HTTP/2 Push 已被主流浏览器废弃，优先使用 preload
```

> 当前更推荐使用 `<link rel="preload">` 替代 Server Push。

#### Brotli 压缩

比 GZIP 压缩率高 15-20%，现代浏览器均支持。服务端配置即可启用（见上文 Nginx 配置）。

#### 模块联邦 (Module Federation)

Webpack 5 特性，实现运行时跨应用共享模块，适合微前端架构：

```js
// webpack.config.js - 宿主应用
new ModuleFederationPlugin({
  name: 'host',
  remotes: {
    remoteApp: 'remoteApp@http://remote.com/remoteEntry.js',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
});
```

优势：避免重复加载共享依赖，各应用独立部署。

#### Navigation API

现代页面导航（SPA 路由切换）不触发传统的 `pageshow`/`pagehide` 事件。Navigation API 提供了统一的导航事件处理。

```js
// 监听导航事件
if ('navigation' in window) {
  window.navigation.addEventListener('navigate', (event) => {
    console.log('Navigating to:', event.destination.url);
    // 可以拦截导航，自定义处理
    // event.intercept({ handler: () => myCustomHandler() });
  });

  window.navigation.addEventListener('navigatesuccess', (event) => {
    console.log('Navigation succeeded');
  });

  window.navigation.addEventListener('navigateerror', (event) => {
    console.log('Navigation failed:', event);
  });
}

// 编程式导航
window.navigation.navigate('/new-page');
```

**与 History API 对比：**
- `history.pushState`：只改变 URL，不触发导航事件
- `Navigation API`：统一的导航生命周期，支持拦截和自定义处理

#### View Transitions API

实现页面/组件级别的过渡动画，无需手动编写 CSS 动画。

```js
// 页面级过渡
document.startViewTransition(() => {
  // 执行 DOM 更新
  document.body.classList.toggle('dark-mode');
});

// CSS 中定义过渡样式
::view-transition-old(root) {
  animation: fade-out 0.3s;
}
::view-transition-new(root) {
  animation: fade-in 0.3s;
}

// 配合 SPA 路由使用
router.on('route-change', () => {
  document.startViewTransition(() => {
    renderNewRoute();
  });
});
```

**适用场景：**
- 主题切换动画
- 列表项重排动画
- 页面路由切换过渡
- 骨架屏到真实内容的过渡

#### Island Architecture (Astro)

Astro 默认不发送任何 JS 到客户端，仅在有交互的组件（Island）上水合：

```astro
---
// Astro 组件，默认静态渲染
const posts = await fetchPosts();
---

<article>
  {posts.map(post => <PostCard post={post} />)}
</article>

<!-- 仅这个组件需要 JS 交互 -->
<InteractiveComments client:load={true} />
```

优势：大幅减少首屏 JS 体积，提升加载性能。

#### Streaming SSR

流式服务端渲染，分块返回 HTML，无需等待全部数据即可开始渲染：

```js
// React 18 Streaming SSR
import { renderToPipeableStream } from 'react-dom/server';

app.get('/', (req, res) => {
  const { pipe } = renderToPipeableStream(<App />, {
    onShellReady() {
      // HTML 骨架就绪，立即开始流式传输
      res.setHeader('Content-type', 'text/html');
      pipe(res);
    },
  });
});
```

优势：用户更快看到页面内容，Suspense 边界外的内容优先渲染。

#### 性能数据上报策略

```js
// 使用 sendBeacon 上报，页面卸载时也不会丢失
function reportPerformance(data) {
  const payload = JSON.stringify({
    url: location.href,
    timestamp: Date.now(),
    ...data,
  });

  if (navigator.sendBeacon) {
    navigator.sendBeacon('/api/perf', payload);
  } else {
    fetch('/api/perf', {
      method: 'POST',
      body: payload,
      keepalive: true, // 替代 sendBeacon
    });
  }
}

// 页面卸载时上报
window.addEventListener('unload', () => {
  const timing = performance.timing;
  reportPerformance({
    dns: timing.domainLookupEnd - timing.domainLookupStart,
    tcp: timing.connectEnd - timing.connectStart,
    ttfb: timing.responseStart - timing.navigationStart,
    domReady: timing.domInteractive - timing.navigationStart,
    load: timing.loadEventEnd - timing.navigationStart,
  });
});
```

> - 采样上报：非 100% 用户上报，降低服务端压力
> - 压缩上报：使用 gzip 压缩 payload
> - 合并上报：批量累积后一次性发送

## 八、节流与防抖

高频事件（scroll、resize、input）中控制回调执行频率：

```js
// 节流：固定时间间隔内最多执行一次
function throttle(fn, delay) {
  let last = 0;
  return function (...args) {
    const now = Date.now();
    if (now - last >= delay) {
      last = now;
      fn.apply(this, args);
    }
  };
}

// 防抖：等待停止触发后执行
function debounce(fn, delay) {
  let timer;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// 使用示例
window.addEventListener('scroll', throttle(handleScroll, 100));
input.addEventListener('input', debounce(search, 300));
```

- **节流**：持续触发期间定期执行，适合 scroll 位置检测
- **防抖**：等待用户停止操作后执行，适合搜索输入

## 九、requestAnimationFrame 动画

使用 `requestAnimationFrame` 实现与显示器刷新率同步的 60fps 动画：

```js
function animate() {
  const progress = (Date.now() - startTime) / duration;
  if (progress < 1) {
    element.style.transform = `translateX(${progress * 100}px)`;
    requestAnimationFrame(animate);
  }
}
requestAnimationFrame(animate);
```

> 优先使用 `transform` 和 `opacity` 做动画，这两个属性走合成层，不触发重排重绘。

## 十、Hybrid / 原生侧优化

#### WKWebView (iOS)

iOS8 起用 WKWebView 替代 UIWebView，性能和内存大幅改善，滚动事件实时触发。

#### 内置 WebView 内核 (Android)

Android 碎片化严重，内置 X5 等高性能内核可统一体验、提升性能。

#### 离线包缓存

下发离线包，将 HTML/CSS/JS 资源预置到客户端，实现页面秒开。

#### 原生代理请求

通过原生网络通道而非 WebView 发起请求，速度更快，可加密封装。

#### 数据预加载

客户端提前请求下一页数据，前端请求时直接取缓存。微信、UC Feed 流等广泛使用。

#### 小程序 / 轻量容器

阉割版 WebView + 强缓存机制，限制能力换取性能。

#### Weex / React Native

JS 驱动 Native 渲染，复杂动画可交由原生执行（如 BindingX），避免频繁 JS-Native 通信。
