## 基础理论

### 背景

随着互联网技术的发展，尤其是Web 2.0、移动互联网、物联
网等技术的普及，传统的磁盘数据库面临以下挑战：

- 高并发访问压力：磁盘I/O成为瓶颈，难以支撑数十万级甚
  至百万级的并发访问
- 实时性要求提高：业务对数据响应速度要求毫秒甚至亚毫秒
  级
- 数据结构日益复杂：越来越多业务需要非关系型的灵活数据
  模型

内存数据库（如Redis、Memcached）应运而生，它们将数据
保存在内存中，读写速度极快，适用于缓存、高速查询、实时分
析等场景

### CAP理论

- C（一致性，Consistency）所有节点在同一时间的数据一致
- A（可用性，Availability）每次请求都能在有限时间内获得响应
- P（分区容忍性，Partition tolerance）系统即使出现网络分区，仍能继续运作。也就是说，某些节点无法通信，系统仍能保持可用性

### BASE思想

- Basically Available（基本可用）系统允许部分功能降级，保证核心功能可用
- Soft state（软状态）系统中的数据可以存在中间状态，不要求实时一致
- Eventually consistent（最终一致性）系统在一段时间后，数据最终会达到一致状态

Redis作为NoSQL代表，采用BASE思想，牺牲强一致性换取高可用性和性能

### RWN理论（副本读写一致性模型）

RWN是描述副本一致性的一种模型：

- R：一次读取操作需要从R个副本读取
- W：一次写入需要写入W个副本
- N：副本总数

Redis通过配置主从复制与RWN原理间接关联：通常"主写从读"，但Redis本身不严格实现RWN，需要上层架构保障。

### NoSQL数据库

NoSQL（Not Only SQL）数据库是非关系型数据库，特点：

- 无模式：无需固定模式（Schema-less），数据结构灵活
- 扩展性：水平扩展能力强，适合大规模分布式系统
- 多样性：支持多种数据结构（如键值、文档、列式、图结构等）

Redis属于NoSQL中的Key-Value存储，支持丰富数据结构、事务、持久化等，功能更强大

## Redis

### 简介

Redis（Remote Dictionary Server）是开源的、基于内存的Key-Value数据库，主要特点如下：

- 所有数据在内存中操作，速度极快
- 支持多种数据类型（String、List、Set、Hash、ZSet等）
- 支持持久化（RDB、AOF）
- 支持主从复制与哨兵高可用。
- 丰富客户端支持（Java、Python、PHP、C、Node.js等）

Redis不仅是缓存工具，更是轻量级的数据结构服务器

### 适用场景

Redis广泛应用于以下业务场景：

| 场景          | 描述                     |
| ------------- | ------------------------ |
| 缓存系统      | 加速读取，降低数据库压力 |
| 排行榜/计数器 | 利用ZSet、String高效实现 |
| 会话存储      | 存储用户Session数据      |
| 消息队列      | 利用List、Pub/Sub实现    |
| 实时数据分析  | 基于数据结构做快速计算   |
| 限流/防刷     | 结合Key过期机制控制频率  |

### 原理

Redis 是一个运行在内存中的高性能 Key-Value 数据库，主要通过以下机制实现其高效运行：

- 内存存储 + 单线程事件模型：所有操作在内存中完成，无需磁盘I/O，单线程避免了上下文切换和加锁开销
- I/O多路复用：使用 epoll 等机制处理高并发连接
- 非阻塞命令执行：即便客户端数量多，响应依然迅速
- 数据结构驱动：内部实现了高效的数据结构如跳表（索引数据结构）、压缩列表、字典等，用于不同类型的键值处理。

Redis不仅仅是Key-Value数据库，更是一个支持多种数据结构的"数据结构服务器"

## API使用

### Key类型

Redis 的 Key 是字符串类型，可以是任何二进制安全的值

| 命令               | 描述                                |
| ------------------ | ----------------------------------- |
| SET key value      | 设置键值                            |
| GET key            | 获取键值                            |
| DEL key            | 删除键                              |
| EXISTS key         | 判断 key 是否存在                   |
| KEYS pattern       | 获取所有匹配 pattern 的 key（慎用） |
| EXPIRE key seconds | 设置 key 过期时间                   |
| TTL key            | 查看剩余时间                        |
| RENAME key newkey  | 重命名 key                          |

### String类型

String 是最基本的类型，二进制安全，一个 String 最大可以存储 512MB

```sql
SET name "Redis"
Get name

INCR counter # 自增DECR
counter # 自减
```

应用场景

1. 网页内容缓存：将频繁访问的网页内容存储在Redis中，减轻数据库负担，提高访问速度
2. Token存储：存储用户登录令牌，实现快速验证和会话管理
3. 验证码：存储短信或邮箱验证码，设置适当的过期时间
4. 计数器：记录网站访问量、文章点赞数等需要频繁更新的计数数据

### Hash类型

Hash 是键值对集合，适合存储对象，每个 Hash 可包含上百万字段

```sql
# 单字段操作
HSET user:1001 name "Tom“
HGET user:1001 name

# 多字段操作
HMSET user:1001 age 30 city "Beijing“
HGETALL user:1001

# 字段管理
HDEL user:1001 age
HEXISTS

# 统计操作
HLEN user:1001
```

应用场景

1. 用户信息存储：Hash结构非常适合存储用户信息，每个字段对应用户的一个属性
2. 对象属性映射：适合存储任何需要映射多个属性的对象数据

### Redis Set 类型和 ZSet 类型

Set 类型（无序集合）：元素唯一、无序，底层使用哈希表

```sql
# 添加元素
SADD tags "java" "redis" "python"

# 查看所有元素
SMEMBERS tags

# 删除元素
SREM tags "java"

# 检查元素
SISMEMBER tags "redis"

# 计算元素数量
SCARD tags
```

应用场景

1. 标签系统：存储文章、商品等的标签，利用Set的唯一性特点自动去重
2. 数据去重：利用Set的唯一性特性，可以轻松实现数据去重功能
3. 抽奖系统：将参与者ID加入Set，使用SRANDMEMBER或SPOP随机抽取
4. 共同好友查询：使用SINTER命令查找两个用户的共同好友。

ZSet 类型（有序集合）：每个元素关联一个分数（score），按照 score 排序

ZSet结合了Set的唯一性和有序性，非常适合需要排序的场景。每个元素都有一个分数，可以根据分数进行排序

应用场景

1. 排行榜：游戏积分排行、销量排行等，分数可以随时更新
2. 热搜榜：根据搜索次数或热度排序的关键词列表
3. 优先队列：根据优先级处理任务，分数代表优先级

### List类型

双端链表结构，支持头尾操作

```sql
LPUSH news "title1" # 左侧添加LPOP news # 左侧弹出
RPUSH news "title2" # 右侧添加RPOP news # 右侧弹出

LRANGE news 0 -1 # 查看所有元素
LLEN news # 获取列表长度
```

应用场景

1. 消息队列：利用List的LPUSH+RPOP（或RPUSH+LPOP）可以实现简单的消息队列，也可以使用BRPOP实现阻塞式队列，消费者等待新消息
2. 时间线和聊天记录：社交媒体的时间线、聊天应用的最近消息等，聊天记录存储

### Redis Key超时机制

- 所有 Key 可设置过期时间（单位：秒）
- 到期后自动删除，不再可访问

```sql
# 设置过期
EXPIRE session:12345 3600 # 1小时
PEXPIRE session:12345 60000 # 毫秒
TTL session:12345 # 查看剩余时间

# 也可以在设置值的同时指定过期时间：
SET token:user:1001 "abc123" EX 1800 # 30分钟

# 永久不过期 默认所有 Key 永久存在，除非主动设置过期或手动删除，取消过期设置：
PERSIST session:12345 # 移除过期时间

# 查看过期状态：
TTL key # 返回-1表示永不过期，-2表示已过期
```

应用场景

1. 验证码：短信验证码、邮箱验证码通常只在短时间内有效
2. 临时缓存：对于一些临时数据，可以设置合适的过期时间，避免手动清理
3. 会话过期：用户登录会话管理，超过一定时间自动失效

## Redis持久化与复制

### Redis持久化

Redis是内存数据库，默认情况下断电数据会丢失，因此提供了两种持久化机制，保障数据恢复能力

- RDB（快照）机制

  RDB（Redis DataBase）是以"快照"的方式将内存中的数据持久化到磁盘上的 .rdb 文件中

  特点：数据压缩后存储，占用空间小，便于备份

  加载优先级：启动时优先加载 .rdb 文件恢复数据

  性能影响：对性能影响较小，适合用于灾难恢复

- AOF（Append Only File）机制

  AOF通过将每次写命令记录在日志中，以命令形式重建数据库

  特点

  - 每次写操作都记录，数据恢复更完整
  - 可设定不同的刷盘策略：always、everysec、no
  - 文件较大，写入频率高时性能略低于RDB

|    特性    |             RDB              |           AOF            |
| :--------: | :--------------------------: | :----------------------: |
|    速度    |          写入性能高          | 稍慢（尤其 always 模式） |
| 数据完整性 | 可能丢失部分数据（间隔触发） |       几乎不丢数据       |
|  文件体积  |             较小             |           较大           |
|  启动速度  |              快              |           略慢           |
|  应用场景  |      数据备份、快速恢复      |       高可靠性需求       |

通常推荐：同时开启AOF + RDB，AOF优先恢复
